## Context

Проект стартует с почти пустого репозитория и подробного ТЗ в `task.md` для Telegram-бота `@sk_fdbck_bot`. Основная ценность — быстро превращать сырые мысли пользователя (текст/голос) в готовый отзыв без длинной анкеты и без потери живого тона. Архитектура должна быть модульной, чтобы контент и prompt-логика менялись без правок Python-кода.

Технические ограничения:
- Runtime: Python + `aiogram 3`.
- Интеграции: Telegram Bot API, AssemblyAI (асинхронная транскрипция), Azure OpenAI (генерация текста).
- Хранилище: SQLite для v1.
- UX-ограничение: диалог не блокируется ожиданием расшифровки.
- Контент-ограничение: гендерно-нейтральные формулировки как в сообщениях бота, так и в итоговом отзыве.

## Goals / Non-Goals

**Goals:**
- Реализовать полный FSM-диалог от приветствия до согласия/отказа на публичную публикацию.
- Поддержать текстовые и голосовые ответы в основных шагах сбора фактуры.
- Обеспечить асинхронную обработку голосовых и объединение транскриптов перед генерацией.
- Генерировать структурированный отзыв через Azure OpenAI и поддерживать неограниченный цикл ручных правок.
- Сохранять данные отзыва в SQLite и отправлять Сергею уведомление с сырыми и финальными данными.
- Вынести тексты и prompts в `content/*.yaml` и `prompts/*.md`.

**Non-Goals:**
- Публичная веб-витрина отзывов.
- Автопостинг в LinkedIn.
- Англоязычная локализация и мультиязычный UX.
- Полноценная админка и аналитический дашборд.
- Миграция на внешнюю БД (Supabase/PostgreSQL) в первой версии.

## Decisions

### 1) Оркестрация через FSM в `aiogram 3`
Используется пошаговая FSM-модель с явными состояниями из ТЗ (`START -> ... -> DONE`), чтобы предсказуемо обрабатывать ветвления и возвраты.

Рассмотренные альтернативы:
- Свободный stateless flow по эвристикам сообщений. Отклонено: сложно управлять пропущенными шагами и edit-циклом.
- Кастомная state-машина поверх Redis. Отклонено для v1: избыточная сложность на старте.

### 2) Асинхронная транскрипция AssemblyAI с job tracking
Голосовые отправляются на транскрипцию сразу при получении, в сессию сохраняется `job_id` и привязка к шагу. Диалог сразу продолжается.

Рассмотренные альтернативы:
- Синхронная блокировка на транскрипции. Отклонено: ломает UX и увеличивает drop-off.
- Локальная speech-to-text библиотека. Отклонено: хуже точность/надёжность и больше операционного шума.

### 3) Генерация и редактирование через два prompt-файла
`prompts/generate_review.md` используется для первичной генерации, `prompts/rephrase_review.md` — для запроса переформулировки (или ручного редактирования пользователем).

Рассмотренные альтернативы:
- Хардкод системного prompt в коде. Отклонено: усложняет тонкие правки контента.
- Один универсальный prompt для всего. Отклонено: хуже управляемость для отдельных режимов.

### 4) Контент как конфиг (`texts.yaml`, `thinking.yaml`)
Все пользовательские сообщения и фразы индикатора "думаю" хранятся вне кода и валидируются при старте.

Рассмотренные альтернативы:
- Inline-строки в обработчиках. Отклонено: невозможно быстро редактировать tone-of-voice.

### 5) SQLite как источник истины v1
Запись создаётся в конце сценария с сохранением сырья (`answers_raw`) и двух версий отзыва (`review_generated`, `review_final`), а также флагов `is_public`, `notified`.

Рассмотренные альтернативы:
- Redis-only persistence. Отклонено: volatile хранилище не закрывает аудит и последующую публикацию.
- Supabase сразу. Отклонено для v1: преждевременная оптимизация при локальном/одиночном использовании.

### 6) Fail-soft стратегия интеграций
Ошибки внешних сервисов не должны терять пользовательский прогресс:
- если транскрипт не готов вовремя, пользователь видит индикатор ожидания;
- если транскрипция недоступна, бот предлагает повторить ответ текстом;
- если LLM-генерация падает, бот запрашивает повторную попытку без сброса шага.

## Risks / Trade-offs

- [Параллельные голосовые ответы приходят в разном порядке] -> Привязка `job_id` к конкретному question key и сортировка при сборке payload.
- [Задержка AssemblyAI увеличивает время до финального отзыва] -> Background polling + индикатор "думаю" + таймаут и fallback на частичную генерацию.
- [LLM может нарушать гендерно-нейтральность] -> Явный constraint в prompt + пост-валидация перед показом пользователю.
- [Потеря FSM-сессии при рестарте процесса с MemoryStorage] -> Для production рекомендовать Redis; для локального v1 оставить MemoryStorage как допустимый режим.
- [Контентные ключи в YAML могут сломаться из-за ручной правки] -> Startup validation обязательных ключей и fail-fast при запуске.

## Migration Plan

1. Добавить структуру директорий `bot/`, `content/`, `prompts/`.
2. Реализовать SQLite-схему и слой доступа (`db.py`) с idempotent-инициализацией.
3. Реализовать FSM и обработчики текст/voice, затем интеграции AssemblyAI и Azure.
4. Подключить загрузку YAML/MD-контента и валидацию конфигов на старте.
5. Прогнать end-to-end сценарий локально и на staging-боте.
6. Деплой на VPS/Railway/Fly.io.

Rollback:
- При критическом сбое отключить webhook/polling бота и откатить на предыдущий релиз.
- SQLite-файл сохраняется как recoverable state; частично заполненные отзывы остаются доступными.
